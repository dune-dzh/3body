Three-body problem — computational solution

The trajectory you see is a numerical solution: Newtonian gravity in the plane, stepped in time on the server and streamed here for visualization — not a closed-form formula (those are rare for n = 3).

What “the solution” is on this page

Solution here means the time-evolving positions of three masses that satisfy Newton’s laws, computed approximately: we cannot write one neat function r(t) for arbitrary three-body setups, so the program integrates the equations of motion step by step. The canvas is that approximate trajectory over time.

Schematic: every pair attracts. The actual run uses the same pairwise idea in 2D with masses and speeds you can change only in code — the UI explains the method, not a parameter panel.

Model & integration (matches the server)

State for each body i: position (x, y) and velocity (vx, vy). Constants in code use G = 1 and a small softening ε so accelerations stay finite when two points get very close.

Acceleration of body i from body j (i ≠ j):

ai←j = G · mj · (rj − ri) / (‖rj − ri‖² + ε)3/2

Sum the contributions from both other bodies, then each step:

v ← v + a·Δt
x ← x + v·Δt

Several small Δt steps are taken between frames you receive over the WebSocket, so the orbit shape is a faithful sample of that discrete solution, not an artistic sketch.

Behind this run

Short answers to “what am I seeing?” and “how is it produced?” — without pretending a bullet list is a physics lesson. For equations and the same model as the server, use Model & integration in the left column.

What you’re watching

The curves are a real-time numerical orbit: the server integrates gravity in the plane and streams positions over a WebSocket. This page draws samples as it receives them — not a pre-rendered loop. If the stream hiccups, the picture reflects that; if it runs, the mess on screen is still the same equations stepping forward in time.

Why it’s tuned for the screen

Constants are chosen for visibility, not the solar system. G = 1 in code units. A small softening ε = 10−2 keeps accelerations from exploding when two bodies skim past each other — standard trick in N-body demos so the timestep can stay reasonable.

How often things update

  • Integration: each substep uses Δt = 0.01. The server does four substeps between messages, so simulation time advances 0.04 per network frame.
  • Updates to you: about 60 frames per second of JSON like { "bodies": [ { "x", "y", "m" }, … ] } — three bodies, same order as red → cyan → lime on the canvas.
  • Integrator style: accelerations from current positions, then velocity and position updates in order (Euler–Cromer–style in code). It’s easy to read in source and cheap to run; it is not built to preserve energy perfectly over long runs.

What the browser adds (cosmetic only)

  • Trails: roughly the last 500 plotted points per body, then older points are dropped so memory stays flat.
  • Disk motion: dots ease toward each new server position for a calmer look. If the OS is set to reduce motion, that easing is turned off and positions snap to each sample — closer to raw data, still not “more accurate physics.”

Why the shape looks “messy” — and what is conserved

Three-body motion is often chaotic: trajectories stay deterministic, but tiny changes in initial data or in the numerics can diverge noticeably over time. That is expected for this model class, not a sign that the WebSocket is corrupting data.

In an ideal continuous system, energy and angular momentum would be fixed along a solution. Discrete stepping only approximates that; combined with softening, you should treat the run as a clear qualitative picture of gravitationally interacting bodies, not a benchmark of long-term conservation.

To try other orbits, masses, or step sizes, change initial_bodies, DT, SOFTENING, or the substep count in app/main.py and restart the server.

Deployment & PUBLIC_WS_URL

The page opens a WebSocket to the same host as the page, or to window.__WS_BASE__ when the server injects it (see build_index_html and env vars PUBLIC_WS_URL, PUBLIC_HOST, PUBLIC_PORT, PUBLIC_TLS in the app). Use that when the HTML is served from a different origin than the simulation API.

Each color is one mass; bright disks are its current place in the numerical solution. Trails are recent past positions — a visible history of the integrated trajectory.
